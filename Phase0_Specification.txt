PHASE 0 SPECIFICATION: SEARCH-DESIGNED TRELLIS CODES WITH NEURAL DECODING
=============================================================================

Author: Abhishek Manjunath
Date: February 15, 2026

EXECUTIVE SUMMARY

• Block coding system with K=256 information bits, rate R=1/2, N=512 coded bits, using BPSK modulation over AWGN channels.

• Channel impairment consists of periodic sinusoidal interference with randomized period (8-32 symbols), phase, and amplitude representing industrial IoT scenarios.

• Trellis encoder uses S=64 states with time-invariant structure, optimized via evolutionary search constrained by non-catastrophic and connectivity requirements.

• Neural decoder implemented as compact GRU architecture with 64 hidden units, consuming approximately 2.1 million operations matching classical baseline.

• Primary baseline is NASA K=7 constraint-length-7 convolutional code (64 states, d_free=10) decoded with mismatched Viterbi ignoring interference.

• Performance evaluated using block error rate across SNR range 0-10 dB and INR range -5 to 15 dB via Monte Carlo simulation.

• Success requires searched trellis plus neural decoder outperforming classical code with mismatched decoder at equal 2.1M operation budget.


1. CODE PARAMETERS

The system operates with BPSK modulation using binary symbols from the set {-1, +1}. Each transmitted block contains K=256 information bits, which are encoded at rate R=1/2 to produce N=512 coded bits for transmission. This blocklength represents a practical trade-off between performance and computational feasibility, being long enough to exhibit meaningful coding gain while remaining short enough for rapid iterative search during trellis optimization. The choice of rate 1/2 provides substantial redundancy for error protection, which is appropriate for initial experiments and allows clear performance differentiation between good and poor code designs.

The trellis encoder is structured as a finite-state machine with S=64 states, labeled from 0 to 63. This state count matches the classical K=7 convolutional code baseline, enabling direct fair comparison at equal trellis complexity. The trellis structure is time-invariant, meaning the state transition function and output mapping remain constant across all time steps. At each encoding step, the encoder accepts a single information bit (k=1) and transitions to a new state while producing n=2 output bits, maintaining the rate 1/2 design. The trellis is terminated by forcing the encoder to return to state 0 at the end of each block, which facilitates both encoding and decoding by establishing known boundary conditions.

Several structural constraints are enforced on candidate trellis designs to ensure decodability and reasonable performance. The trellis must be non-catastrophic, meaning it cannot contain cycles that produce zero output weight, as such structures would make certain error patterns undetectable. All states must be reachable from the initial state, and it must be possible to reach the terminal state from any intermediate state, ensuring the trellis is fully connected. Additionally, we target a minimum free distance of at least d_free=8, which provides strong error protection capability. This target is approximately 80% of the baseline K=7 convolutional code's proven d_free=10, representing a code that can correct up to 3 bit errors compared to the baseline's 4-bit correction capability. The slightly lower distance target is reasonable for evolutionary search while still maintaining competitive error correction strength.


2. CHANNEL MODEL

The base channel consists of additive white Gaussian noise with independent identically distributed samples n[t] drawn from a zero-mean Gaussian distribution with variance σ². The signal-to-noise ratio ranges from 0 dB to 10 dB, with particular focus on the operating point of SNR=5 dB as representative of typical communication scenarios. The noise variance relates to SNR through σ²=E_s/SNR_linear, where E_s=1 represents unit symbol energy for BPSK modulation, and SNR_linear=10^(SNR_dB/10) converts the decibel value to linear scale.

The key distinguishing feature of our channel model is the presence of periodic sinusoidal interference, which represents structured impairments commonly encountered in real-world systems. At each time step t, the received signal is r[t]=x[t]+n[t]+i[t], where x[t] is the transmitted BPSK symbol, n[t] is the Gaussian noise, and i[t]=A·sin(2πt/P+φ) is the periodic interference component. The interference period P is drawn uniformly from the discrete range [8, 32] symbols, ensuring that multiple complete interference cycles occur within each block of 512 symbols. The phase offset φ is drawn uniformly from [0, 2π] at the start of each block, making the interference unpredictable to the decoder. The interference amplitude A is determined by the interference-to-noise ratio through the relationship A=√(2·INR_linear·σ²), where INR_linear=10^(INR_dB/10). This formula accounts for the fact that a sinusoid with amplitude A has average power A²/2, so the interference-to-noise ratio becomes INR=(A²/2)/σ².

During neural decoder training, we randomize the channel parameters to encourage robustness. The SNR is varied uniformly over [0, 10] dB, the INR is varied uniformly over [-5, 10] dB representing moderate to strong interference conditions, the period P is randomized over [8, 32] symbols, and the phase φ is randomized over [0, 2π] for each block. This randomization ensures the decoder cannot simply memorize specific interference patterns but must learn general principles for robust decoding.

To evaluate generalization and robustness, we test the trained system on out-of-distribution conditions including shifted INR values in the range [10, 15] dB that are stronger than any interference seen during training, and shifted periods P in {6, 7, 33, 34} that fall outside the training range. The decoder is assumed to know the noise variance σ² (which could be estimated from pilot symbols or preamble sequences in practice) but does not know the interference parameters P, φ, or A. This represents a realistic mismatch scenario where the receiver has imperfect channel knowledge, which is central to the project's motivation.

The periodic interference model is justified by real-world communication scenarios. Industrial IoT deployments often experience periodic electromagnetic interference from motors, machinery, and rotating equipment. Mixed-signal systems encounter clock interference from switching power supplies and digital circuitry. These interference sources exhibit the periodic structure captured by our sinusoidal model, making this a practically relevant channel impairment rather than an artificial academic construct.


3. PERFORMANCE METRICS

The primary performance metric is block error rate (BLER), defined as the probability that a decoded block differs from the transmitted block in at least one bit position. Formally, BLER is computed as the number of blocks containing one or more bit errors divided by the total number of transmitted blocks. Our target performance level is BLER ≤ 10^(-3) at the operating point of SNR=5 dB and INR=5 dB, representing one block error per thousand transmissions under moderate interference conditions.

Performance evaluation sweeps across multiple SNR values from the set {0, 2, 4, 6, 8, 10} dB, providing six data points to characterize the full operating range. At each SNR value, we further vary the interference strength across INR values {-5, 0, 5, 10} dB to understand how performance degrades as interference becomes more severe. Results are typically presented as BLER versus SNR curves with separate curves for each INR level, allowing visual comparison of how different methods handle varying interference conditions.

Monte Carlo simulation is used to estimate BLER through empirical averaging. During the trellis search phase (Phase 3), quick fitness evaluations use 1,000 randomly generated blocks per SNR point to provide rapid feedback for the evolutionary algorithm. For final evaluation (Phase 4), we increase to 100,000 blocks per SNR point to achieve statistical confidence. At BLER=10^(-3), simulating 100,000 blocks yields approximately 100 error events, providing a 95% confidence interval of roughly ±6% around the estimated error rate, which is adequate for meaningful performance comparison.

Beyond the primary BLER metric, we also track bit error rate (BER) as a secondary measure, representing the fraction of individual bit errors rather than block errors. The relationship between computational complexity and error rate is critical, so we construct Pareto curves showing BLER versus operation count to visualize efficiency trade-offs. Most importantly, we quantify robustness by measuring the BLER degradation when interference parameters shift outside the training distribution, with successful designs showing graceful degradation rather than catastrophic failure.


4. COMPUTATIONAL CONSTRAINTS

Our baseline reference decoder is the Viterbi algorithm operating on a 64-state trellis, which requires approximately S²×N operations where S=64 states and N=512 time steps. This yields 64²×512=4,096×512=2,097,152 operations, which we round to 2.1M operations for reference. Each operation is defined as one floating-point multiply-accumulate (MAC) or equivalently one FLOP, providing a hardware-independent measure of computational complexity. The Viterbi algorithm performs add-compare-select operations at each trellis stage, with the number of computations per stage scaling quadratically with the state count because each of the S states must consider transitions from potentially S predecessor states.

The neural decoder is allocated a budget matching the baseline at approximately 2.1 million operations, demonstrating that joint code-decoder optimization can achieve competitive performance at equal computational cost rather than requiring additional resources. We define operations for neural networks by counting matrix multiply-accumulate operations in fully connected and recurrent layers, element-wise operations in activation functions such as ReLU or tanh, and normalization operations if batch normalization or layer normalization is employed. Tools such as pytorch-OpCounter or thop are used to verify operation counts for implemented neural architectures.

When comparing methods with different computational budgets, fairness requires that all baselines receive equal computational resources. Since our neural decoder uses the same 2.1M operation budget as the 64-state Viterbi baseline, we ensure direct comparison at matched complexity. If exploration reveals that neural architectures require different operation budgets, classical baselines would be adjusted accordingly, either by modifying trellis size, running multiple iterations, or implementing list decoding with appropriate list sizes. This fairness principle ensures we compare the best use of a given computational budget rather than allowing any method unlimited resources.


5. BASELINE METHODS

Baseline 1 serves as the primary comparison and consists of a standard rate-1/2 constraint-length-7 convolutional code with generator polynomials (133, 171) in octal notation, which is the NASA standard convolutional code widely used in satellite communications. This code has 2^(K-1)=2^6=64 states and encodes K=256 information bits to produce N=512 coded bits, matching our blocklength exactly. The code is terminated by driving the encoder to the all-zeros state. This code has a proven minimum free distance of d_free=10, well-documented in coding theory literature. Decoding is performed using the Viterbi algorithm with branch metrics computed under the assumption of AWGN-only channel conditions, meaning the decoder computes metric[t]=|r[t]-x[t]|² which ignores the interference component i[t]. This represents a realistic mismatched decoder scenario where the receiver model does not match the actual channel conditions. The computational complexity is 64²×512=2,097,152 operations, approximately 2.1 million operations. The purpose of this baseline is to show the performance of a well-established classical code when decoded with model mismatch, representing the standard approach that does not account for structured interference.

Baseline 2 uses the same constraint-length-7 convolutional code as Baseline 1 but employs an oracle Viterbi decoder that has perfect knowledge of the interference parameters. The branch metrics are computed as metric[t]=|r[t]-x[t]-i[t]|² where i[t]=A·sin(2πt/P+φ) is calculated using the true values of A, P, and φ. This oracle decoder is unrealistic in practice because the receiver would not know these parameters perfectly, but it serves as an upper bound on performance achievable by matched decoding with classical codes. The computational complexity remains 2.1 million operations, identical to Baseline 1. Comparing Baselines 1 and 2 directly quantifies the performance gap caused by model mismatch, which is a central theme of this project.

Baseline 3 is an ablation study that isolates the contribution of trellis search from the contribution of neural decoding. This baseline uses a randomly generated 64-state rate-1/2 trellis encoding K=256 information bits to N=512 coded bits. The random trellis is generated to satisfy basic structural constraints such as being non-catastrophic, fully connected, and having minimum free distance d_free≥5, but it is not optimized through evolutionary search. Critically, this random trellis is decoded using exactly the same neural decoder architecture and training procedure as our proposed method. The computational complexity matches our proposed system at approximately 2.1 million operations. If our searched trellis significantly outperforms this random trellis baseline while using the same decoder, it provides evidence that the trellis search is finding meaningful structure. Conversely, if performance is similar, it suggests the search is not effectively discovering better codes.

Baseline 4 is an reference point that provides context by comparing against modern coding techniques. This baseline uses a CRC-aided polar code constructed for K=256 information bits plus an 11-bit CRC for error detection, yielding 267 total information bits encoded to N=512 coded bits at approximately rate 1/2. The polar code construction follows the 3GPP 5G standard construction if available, or otherwise uses Arikan's original construction method. Decoding employs successive cancellation list (SCL) decoding with list size L=4, which is intentionally kept small to maintain fair computational comparison. The decoder uses mismatched metrics assuming AWGN-only conditions and ignoring interference. Polar code complexity scales as approximately L×N×log₂(N)=4×512×9≈18,432 operations, which is much cheaper than Viterbi due to the fundamentally different decoding algorithm structure. This baseline shows state-of-the-art performance from modern codes and provides important context for reviewers, but it is not the primary comparison target because polar codes belong to a different code family with different decoding paradigms. The main story focuses on trellis-based comparisons in Baselines 1-3.


6. SUCCESS CRITERIA

Success for Phase 2 (neural decoder validation) is achieved if the neural decoder beats the mismatched Viterbi baseline (Baseline 1) by at least 0.5 dB at BLER=10^(-3), demonstrating that learned decoding can compensate for model mismatch. This 0.5 dB threshold represents a meaningful but achievable performance gain, corresponding to approximately 12% reduction in required transmit power. The threshold is clearly distinguishable from Monte Carlo estimation noise with 100,000 trials, which provides approximately 0.2 dB confidence intervals. Smaller thresholds like 0.1-0.2 dB would be difficult to measure reliably, while larger thresholds like 2-3 dB would be unrealistic given that classical codes with mismatched decoding already perform reasonably well. The training process must be stable with loss functions decreasing over epochs, and the trained decoder must generalize to interference parameters outside the training distribution without catastrophic performance collapse.

Success for Phase 3 (trellis search) requires that the searched trellis outperforms a random trellis (Baseline 3) by at least 1 dB at BLER=10^(-3) when both are decoded with the same neural architecture, providing evidence that evolutionary search discovers meaningful code structure. The evolutionary algorithm should show clear fitness improvement over generations rather than random walk behavior, and the final discovered trellis must satisfy all structural constraints including being non-catastrophic and fully connected.

The primary success criterion for Phase 4 (final evaluation) is that our complete system, consisting of the searched 64-state trellis paired with the neural decoder, achieves lower BLER than Baseline 1 (classical 64-state convolutional code with mismatched Viterbi) when both methods operate at equal 2.1M operation computational complexity. This demonstrates that joint co-design of code and decoder yields practical performance benefits under realistic mismatch conditions at matched trellis complexity.

Stretch goals that would strengthen the contribution include demonstrating robustness with less than 1 dB performance degradation when INR increases by 5 dB beyond training levels, showing computational efficiency by achieving competitive performance with a 64-state classical decoder while using only one-fourth of the operations, and demonstrating generalization by achieving good performance across both rate 1/2 and rate 2/3 configurations without retraining.

We explicitly identify failure modes to avoid. The project would be considered unsuccessful if the neural decoder performs worse than the mismatched classical baseline, which would contradict the core thesis that learning can compensate for mismatch. If the searched trellis performs equivalently to a random trellis, it indicates the search algorithm is not functioning properly. If the system only works at a single specific SNR/INR operating point and fails elsewhere, it suggests overfitting rather than learning general robust decoding principles. Finally, significantly exceeding the computational budget without corresponding performance gains would undermine the practical relevance of the approach.


7. IMPLEMENTATION PLAN

The project implementation uses Python 3.10 or later with PyTorch 2.0 or later for neural network implementation and training. NumPy and SciPy provide numerical simulation capabilities for channel modeling and Monte Carlo evaluation. Visualization uses Matplotlib and Seaborn for generating performance curves and analysis plots. Operation counting relies on pytorch-OpCounter or thop libraries to verify that neural architectures satisfy computational budgets. All code is version-controlled using Git to track development and enable reproducibility.

The codebase is organized into modular components. The channel.py module implements the channel simulator including AWGN generation, sinusoidal interference generation with configurable period P, phase φ, and amplitude A, and combined channel functions that produce received signals. The trellis.py module defines the trellis code class with methods to specify transition tables and output labelings, an encode function that maps information bits to coded bits, constraint validation to check non-catastrophic and connectivity properties, and loaders for standard codes such as the K=7 convolutional code. The decoders.py module implements classical decoding algorithms including Viterbi with both mismatched AWGN-only metrics and oracle metrics with perfect interference knowledge, with optional BCJR implementation for soft-output decoding if needed. The neural_decoder.py module contains the GRU-based neural architecture with 64 hidden units matching the 2.1M operation budget, training loops with loss computation and optimization, and operation counting utilities. The eval.py module provides evaluation infrastructure including Monte Carlo BLER estimation across multiple SNR and INR values and plotting utilities for generating publication-quality figures. The baselines.py module wraps baseline implementations, potentially interfacing with external libraries for polar code decoding.

Phase 3 introduces additional modules for evolutionary search. The search.py module manages the evolutionary algorithm including population initialization and management, mutation operators that modify trellis transition tables and output labels, crossover operators for combining parent trellises, selection mechanisms such as tournament selection, and constraint checking to ensure all candidate trellises satisfy structural requirements. The fitness.py module implements the two-tier fitness evaluation where cheap evaluations use 1,000 Monte Carlo trials at a single SNR/INR point for rapid iteration, and expensive evaluations use 10,000 trials across multiple SNR values with neural decoder retraining for periodic in-depth assessment.

Validation testing ensures implementation correctness. Unit tests verify that encoding followed by decoding in a noiseless channel recovers the original information bits exactly. Smoke tests confirm that AWGN-only performance matches theoretical predictions for classical codes, validating the simulation framework. Sanity tests verify that the oracle decoder (Baseline 2) always outperforms the mismatched decoder (Baseline 1) as expected, since perfect channel knowledge should never hurt performance.


8. RISK ANALYSIS

Several risks could impact project success, each with mitigation strategies. The first significant risk is that the neural decoder may fail to outperform classical decoding, which would be medium probability and high impact as it undermines the core thesis. Mitigation strategies include starting with proven GRU architectures documented in literature rather than novel untested designs, conducting extensive hyperparameter tuning across learning rates, hidden sizes, and regularization strengths, ensuring training data thoroughly covers the full parameter distribution to avoid blind spots, and adopting a fallback position that claims robustness and graceful degradation rather than raw performance superiority if absolute BLER values are not competitive.

The second major risk is that evolutionary search may not discover good trellises despite many generations, which is medium probability and high impact as it represents the core novelty. Mitigation approaches include initializing the population by perturbing known good codes such as classical convolutional codes rather than pure random initialization, running multiple independent search runs with different random seeds to avoid local optima, experimenting with different search algorithms including simulated annealing or gradient-based relaxations if evolutionary approaches stall, and recognizing that analyzing why search failed can itself be a publishable contribution to understanding code design landscapes.

The computational budget being too restrictive is low probability but medium impact. Mitigation is straightforward since we control the budget definition and can adjust it if initial experiments show infeasibility, we prioritize operation count over wall-clock time so hardware variations are less critical, and GPU acceleration can be applied to neural decoder components if CPU-based approaches prove too slow.

Baseline methods dominating our approach is medium-to-high probability with medium impact. Mitigation involves carefully choosing baselines to reflect realistic mismatch rather than idealized perfect channel knowledge, focusing the narrative on robustness and graceful degradation rather than competing purely on AWGN performance where classical codes already excel, and emphasizing value in specific high-interference regimes and parameter shift scenarios where classical methods are known to struggle.

Inability to reproduce documented baseline performance is low probability but high impact as it damages credibility. Mitigation includes using well-documented standard codes such as the K=7 NASA convolutional code with published performance data, validating on AWGN-only channels first where theoretical bounds exist, and starting from reference implementations when available rather than coding from scratch.

Training instability with neural networks is medium probability and medium impact. Standard mitigation techniques include gradient clipping to prevent exploding gradients, learning rate schedules that reduce step size as training progresses, regularization through dropout and weight decay to prevent overfitting, careful weight initialization using methods appropriate for recurrent architectures, and close monitoring of training curves to detect divergence early.
